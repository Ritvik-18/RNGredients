## Problem Statement

Home cooks face three compounding friction points:

1. **Decision fatigue** — "What do I cook today?" leads to paralysis or defaulting to delivery.
2. **Grocery waste** — avg. US household wastes ~31% of food; items expire before use.
3. **Time inefficiency** — meals cooked individually instead of batch-prepped in optimal order.

RNGredients attacks all three using game mechanics: gacha roll removes the choice burden, decay timers make waste emotionally salient via loss aversion, and CPM scheduling optimizes the prep sequence automatically.

---

## What Makes This Cool

The gacha roll mechanic is the hook. Every competitor (Mealime, PlateJoy, Meal Genius) adds gamification AS a reward layer — badges, streaks, Life Scores. RNGredients uses the game mechanic AS the decision itself. You don't get rewarded for cooking. You roll to find out what you're cooking. That's variable ratio reinforcement (Skinner box) applied to meal selection — and nothing in the existing market does this.

The decay timer reframes food waste as loss aversion, not guilt. Seeing your chicken breast at 20% HP is psychologically different from seeing "expires Apr 21." Users feel compelled to use it. That's the IKEA Effect + Kahneman working together.

The slot machine window settling on "✦ Lemon Herb Chicken ✦" with a rarity flash is the demo moment. A judge who watches that in real time will remember it.

## Premises

1. **The gacha roll is the right demo centerpiece.** Variable ratio reinforcement is the strongest psychological hook. Nothing else in the market does this for meal selection. This premise holds.

2. **Skip CPM scheduling for the hackathon.** Judges won't notice the diff between a CPM-optimized queue and a flat ordered list in a 2-min demo. The inventory → roll → quest log loop is the complete story. CPM is Sprint 5 polish.

3. **Full inventory input is worth the build time.** The decay bar on a real grocery item the user typed in is 10x more emotionally resonant than demo data. The "whoa" is doubled when it's your actual chicken breast expiring.

4. **Spoonacular API is correct.** 365k recipes. Ingredient parsing. Dietary filters built in. Own recipe DB is weeks of work that doesn't win a hackathon.

5. **React + FastAPI is the right split.** The gacha animation needs Framer Motion — that requires a real React frontend. The loot table weighting logic (decay urgency × inventory match × stat alignment) is cleaner in Python than in JS. Don't collapse the stack.

## Approaches Considered

### Approach A: Frontend-Only (No Backend)
Call Spoonacular directly from React. Run loot table logic in JS. Use localStorage for inventory. No FastAPI.

**Pros:** Faster to demo. No backend to deploy.
**Cons:** Loot table logic in JS is messier. No persistence. Decay timer math gets fiddly. Harder to extend.
Completeness: 5/10

### Approach B: Full-Stack React + FastAPI (Recommended)
Inventory CRUD in FastAPI + PostgreSQL. Loot table as a Python endpoint. Spoonacular called from backend (hides API key). React gets clean JSON and does the animation.

**Pros:** Clean separation. Python is better for the weighted RNG math. API key secured. Extends into post-hackathon.
**Cons:** More to deploy. Slightly more setup.
Completeness: 8/10

## Recommended Approach

**Approach B — Full-Stack React + FastAPI.**

The loot table weighting is numerical Python work, not presentation work. Keep it in FastAPI. React's only job is the animation and the UI state. Deploy to Railway (backend + PostgreSQL) + Vercel (frontend) — assume judges need a live URL.

The split also keeps your Spoonacular API key out of the client bundle.

### Auth (simplified for hackathon)

No OAuth. Character creation = implicit account creation:
- User enters `username` on `/character-creation` → `POST /api/player` creates the record, returns `player_id` stored in localStorage.
- No password. Guest-mode. No session tokens. For a hackathon demo this is sufficient.
- `player_id` is a UUID sent with every API request. Trust the client (no auth middleware needed for demo).

### Rarity assignment

Rarity is assigned when a recipe is cached from Spoonacular. Rule:
```python
def assign_rarity(nutrition: dict) -> str:
    protein = nutrition.get("protein", 0)
    if protein >= 35:    return "legendary"   # gold flash
    if protein >= 25:    return "rare"        # silver shimmer
    if protein >= 15:    return "uncommon"    # blue tint
    return "common"                            # gray shimmer
```

Simple, deterministic, and nutrition-aligned. Users get rewarded for high-protein rolls.

### Inventory decay formula

Frontend calculates HP% locally — no API call needed:
```js
// expire_date: "2026-04-22", today: "2026-04-19"
const days = Math.max(0, differenceInDays(expire_date, today))  // clamp to 0 minimum
const hp = Math.min(1.0, days / 7)  // cap at 100%
// If expire_date has passed: hp = 0, item shows as "EXPIRED" grayed out
// Under 20%: critical (striped bar), under 50%: warning (dim), over 50%: healthy (solid)
```

Expired items are not deleted automatically — user manually removes them. This is intentional: seeing expired items creates regret that reinforces the game loop.

### MVP inventory form

Two fields only:
- `item_name` (text, required, autocomplete from common groceries)
- `expire_date` (date picker, required, min = today)

Quantity defaults to 1 unit. Category is auto-inferred from name lookup table (chicken → protein, rice → carb, etc.) or defaults to "other." No validation complexity beyond non-empty name + expire_date >= today.

```python
REVERSE_MAP = {}
for canonical, aliases in SYNONYMS.items():
    for alias in aliases:
        REVERSE_MAP[alias.lower()] = canonical

def normalize(name: str) -> str:
    return REVERSE_MAP.get(name.lower(), name.lower())
```

Both user input (inventory) and Spoonacular ingredients use `normalize()` before comparison.

### Loot table — pseudocode

```python
def score_recipe(recipe: Recipe, inventory: list[InventoryItem], goals: dict) -> float:
    inventory_normalized = {normalize(i.item_name) for i in inventory}
    
    # decay_urgency: does this recipe use items about to expire?
    urgency_items = [i for i in inventory if normalize(i.item_name) in
                     {normalize(ing) for ing in recipe.ingredients}]
    decay_urgency = sum(max(0, 1 - (i.expire_date - date.today()).days / 7)
                        for i in urgency_items)
    decay_urgency = min(100, decay_urgency * 25)  # normalize to 0-100

    # inventory_match: % of recipe ingredients already owned
    matched = sum(1 for ing in recipe.ingredients
                  if normalize(ing) in inventory_normalized)
    inventory_match = (matched / max(1, len(recipe.ingredients))) * 100  # 0-100

    # stat_alignment: how close is recipe protein to daily goal / 3 meals?
    target = goals.get("protein", 120) / 3  # default 120g/day = 40g/meal
    stat_alignment = max(0, 100 - abs(recipe.nutrition["protein"] - target) * 3)
    # Calibration: 3 multiplier means ±33g diff = 0 alignment. Adjust if too harsh.

    return decay_urgency * 0.5 + inventory_match * 0.3 + stat_alignment * 0.2


def roll(player_id: uuid.UUID, goals: dict) -> dict:
    inventory = get_inventory(player_id)
    player = get_player(player_id)
    selected = {}
    
    for meal_type in ["breakfast", "lunch", "dinner"]:
        pool = filter_recipes(meal_type, player.dietary_restrictions)
        if len(pool) < 3:
            # Relax: return whatever is available (N < 3)
            pool = get_all_recipes(meal_type)  # ignore dietary filter as fallback
        
        # Score all, pick randomly from top 20% — preserves RNG feel
        scored = sorted(pool, key=lambda r: score_recipe(r, inventory, goals), reverse=True)
        top_pool = scored[:max(5, len(scored) // 5)]
        scores = [score_recipe(r, inventory, goals) for r in top_pool]
        selected[meal_type] = random.choices(top_pool, weights=scores, k=1)[0]
    
    return selected
```

**Test before demo:** Roll with inventory `[chicken exp=tomorrow, rice exp=May 12, lemon exp=Apr 22]`. Expect: dinner recipes using chicken score top 3. If not, increase `decay_urgency` weight to 0.6.

### Error handling

| Scenario | Behavior |
|---|---|
| Spoonacular down | Cache-only mode, no API calls during demo. Non-issue. |
| Recipe cache < 3 per meal type | Show "Roll unavailable — refilling tavern..." with a retry button. Seed script prevents this. |
| expire_date in the past | Form validation rejects it: "Date must be in the future." |
| Roll returns < 3 recipes after filters | Show N recipes available, display "Tavern is sparse — consider relaxing your filters." |
| Re-rolls exhausted | Roll button grays out, shows "0 re-rolls remaining. Resets at midnight." |

   ```json
   [
     {"time_offset_min": 0, "task": "Boil rice", "used_in": ["lunch", "dinner"], "passive": true},
   ```
